# NVIDIA GB200

## Role in this repo

GB200 is the first Blackwell rack-scale target in this hardware set. It matters
for kernels that need Blackwell Tensor Core formats, very large NVLink domains,
and multi-GPU execution across a Grace Blackwell system.

## CUDA target

| Field | Value |
| --- | --- |
| Architecture | NVIDIA Blackwell |
| Compute capability | 10.0 |
| `nvcc` arch | `sm_100` or `sm_100a` when using Blackwell architecture-specific features |
| Common forms | GB200 Grace Blackwell Superchip, GB200 NVL72 rack |
| Primary workload fit | FP4/FP6/FP8 tensor kernels, LLM inference, large distributed training, NVLink-scale benchmarks |

## Useful specs

| Spec | GB200 Grace Blackwell Superchip | GB200 NVL72 |
| --- | ---: | ---: |
| Configuration | 1 Grace CPU, 2 Blackwell GPUs | 36 Grace CPUs, 72 Blackwell GPUs |
| GPU memory and bandwidth | 372 GB HBM3E, 16 TB/s | 13.4 TB HBM3E, 576 TB/s |
| CPU memory and bandwidth | Up to 480 GB LPDDR5X, up to 512 GB/s | 17 TB LPDDR5X, 14 TB/s |
| CPU cores | 72 Arm Neoverse V2 cores | 2,592 Arm Neoverse V2 cores |
| NVLink bandwidth | 3.6 TB/s | 130 TB/s |
| NVFP4 Tensor Core | 40 PFLOPS sparse, 20 PFLOPS dense | 1,440 PFLOPS sparse, 720 PFLOPS dense |
| FP8/FP6 Tensor Core | 20 PFLOPS sparse | 720 PFLOPS sparse |
| FP16/BF16 Tensor Core | 10 PFLOPS sparse | 360 PFLOPS sparse |
| TF32 Tensor Core | 5 PFLOPS sparse | 180 PFLOPS sparse |
| FP64 / FP64 Tensor Core | 80 TFLOPS | 2,880 TFLOPS |

## Kernel-development notes

- Use GB200 for Blackwell-specific kernels only after keeping a portable path
  for older architectures when practical.
- Prefer `sm_100` for broad GB200 builds. Use `sm_100a` only for kernels that
  explicitly require architecture-specific Blackwell behavior.
- Add benchmark metadata for GPU count, NVLink domain, CPU affinity, and
  process placement. On NVL72, topology is part of the benchmark result.
- Blackwell adds lower-precision AI formats that should be isolated in tests:
  NVFP4, FP6, FP8, FP16/BF16, TF32, and FP64.
- Rack-scale memory and interconnect bandwidth are large enough that single-GPU
  measurements can be misleading for production LLM workloads.

## Benchmark priorities

- FP4/FP6/FP8 GEMM and attention kernels.
- KV-cache bandwidth, paged attention, fused normalization, and decode-time
  kernels.
- Multi-GPU all-reduce, all-gather, reduce-scatter, and peer bandwidth.
- Grace CPU to Blackwell GPU transfer and unified workflow measurements.
- Strong-scaling and weak-scaling tests across multiple GPUs.
