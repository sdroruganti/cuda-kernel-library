# NVIDIA RTX A4500

## Role in this repo

RTX A4500 is the local development GPU for this CUDA kernel library. It should
be the first target for correctness checks, build-system validation, profiling
practice, and small-to-medium benchmarks before moving kernels to data-center
GPUs such as A100, H200, GB200, or GB300.

## CUDA target

| Field | Value |
| --- | --- |
| Architecture | NVIDIA Ampere |
| Compute capability | 8.6 |
| `nvcc` arch | `sm_86` |
| Common form | Workstation PCIe card |
| Primary workload fit | Local CUDA development, profiling, memory-bandwidth kernels, tensor-core experiments, PyTorch extension testing |

## Kernel-development notes

- This is the repo's default practical target because the Makefile already uses
  `ARCH ?= sm_86`.
- Use `-arch=sm_86` for native RTX A4500 builds.
- Keep problem sizes configurable. A4500 has much less bandwidth and memory
  capacity than HBM data-center GPUs, but it is large enough to catch most
  correctness, launch-configuration, and indexing issues.
- Do not assume A100 behavior just because both GPUs are Ampere. A4500 is
  compute capability 8.6, while A100 is compute capability 8.0.
- ECC, display use, desktop workloads, thermals, and power limits can affect
  benchmark stability. Record `nvidia-smi` output for any result worth keeping.
- NVLink is available only in a two-GPU A4500 setup with the bridge and software
  support. Single-card development should assume PCIe host/device transfers.

## Benchmark priorities

- Sanity and correctness tests for every kernel.
- SAXPY, copy, transpose, reduction, scan, histogram, stencil, and convolution
  microbenchmarks.
- Occupancy experiments around block size, register pressure, and shared-memory
  usage.
- FP32 and TF32/FP16 Tensor Core experiments where the kernel uses compatible
  math paths.
- PyTorch extension build and integration tests.
- Nsight Systems and Nsight Compute profiling practice on kernels that will
  later be moved to data-center targets.
